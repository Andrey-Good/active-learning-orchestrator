# W20 - Fix Benchmark Label Leakage Through IDs

## Relation To Overall Task
R29 reviewed W19 and confirmed strict JSON is fixed, but found a remaining blocking label leakage path: acquisition-visible `sample_id` and `group_id` contain true label strings. This must be fixed before the benchmark harness can be used for SDK optimization.

## Assumptions And Resolved Ambiguities
- R29 finding is accepted as valid.
- Acquisition-visible identifiers must not encode true labels.
- It is acceptable for generated artifacts to report labels after selection/evaluation, but those labels must come from benchmark-private maps, not acquisition-visible `DataSample` fields.
- Existing smoke artifacts must be regenerated after the fix.

## Goal And Expected Result
Make all IDs and group IDs passed to `SelectionContext`, `StrategyScheduler`, SDK strategies, and `DataSample` opaque with respect to true labels. Keep private dataset metadata capable of mapping opaque IDs back to labels for training, evaluation, diagnostics, and human-readable artifacts.

## In Scope
- Replace label-bearing `sample_id` patterns with opaque IDs such as dataset-prefix + split + integer index that do not include class names.
- Replace label-bearing `group_id` patterns with opaque group IDs that do not include class names.
- Preserve deterministic grouping behavior for duplicate/concentration diagnostics.
- Keep true label only in `BenchmarkSample.label` and benchmark-private mappings.
- Regenerate smoke artifacts.

- Do not mutate SDK private state.

- Smoke run succeeds and artifacts regenerate.
